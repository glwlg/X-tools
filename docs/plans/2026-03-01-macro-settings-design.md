# 宏管理（V2 命令链）设计文档

- 日期：2026-03-01
- 主题：在设置页支持新增/编辑/删除宏，并以命令链方式执行
- 状态：已评审通过（用户确认）

## 1. 背景与现状

当前项目已有“工作流宏”能力，但数据写死在 `src/plugins/workflow_tool.py` 中，设置页 `src/ui/settings_window.py` 没有宏管理入口。

现状问题：

1. 用户无法在设置中新增宏。
2. 宏不可配置、不可扩展、不可持久化。
3. 每次新增宏都需要改代码并重新发布。

## 2. 目标与非目标

### 2.1 目标

1. 在设置页新增“宏”管理页，支持新增、编辑、删除宏。
2. 宏支持 V2 自定义命令链（多步执行）。
3. 宏配置持久化到现有 `config.json`。
4. 保持现有搜索交互：继续通过 `wf`/`workflow`/`flow` 触发执行。
5. 执行结束后将最终结果复制到剪贴板，保持现有用户心智。

### 2.2 非目标

1. 本期不做可视化拖拽流程编排器。
2. 本期不做跨会话变量/条件分支/循环。
3. 本期不改动插件系统总体架构（仅在 workflow 插件内新增执行引擎）。

## 3. 总体方案

采用“配置驱动 + 运行时执行引擎”的方案：

1. 在配置中新增 `workflows` 字段保存宏数据。
2. `WorkflowPlugin` 从配置中读取宏，而非硬编码列表。
3. 设置页新增“宏”子页负责 CRUD 与校验。
4. 执行入口仍是 `workflow_tool.handle_action(workflow_id)`，搜索页逻辑不变。

该方案兼容现有架构，改动集中，扩展性足够。

## 4. 数据模型设计

在 `DEFAULT_CONFIG` 中新增：

```json
{
  "workflows": [
    { "id": "clip-md5", "name": "...", "steps": [ ... ] },
    { "id": "clip-url-encode", "name": "...", "steps": [ ... ] },
    { "id": "clip-base64-encode", "name": "...", "steps": [ ... ] },
    { "id": "now-timestamp", "name": "...", "steps": [ ... ] }
  ]
}
```

单个宏结构：

```json
{
  "id": "clip-url-md5",
  "name": "剪贴板 -> URL编码 -> MD5",
  "description": "把剪贴板文本先编码再做MD5",
  "steps": [
    { "command": "url {clipboard}", "pick": "编码结果" },
    { "command": "hash {prev}", "pick": "MD5" }
  ]
}
```

字段约束：

1. `id`：必填，唯一，小写字母/数字/连字符（正则建议 `^[a-z0-9-]+$`）。
2. `name`：必填，用户可读。
3. `description`：可选，展示在搜索预览中。
4. `steps`：至少 1 步。
5. `step.command`：必填，完整命令文本。
6. `step.pick`：可选，结果选择规则（按结果 `name` 前缀匹配）。

## 5. 命令链语法与变量

每个 step 使用插件命令文本：`<keyword> <args...>`。

支持变量（首版）：

1. `{clipboard}`：当前剪贴板文本。
2. `{prev}`：上一步选中结果的 `path`。
3. `{input}`：运行输入变量，可通过 `wf <workflow-id> <input>` 传入。

解析流程：

1. 对 `step.command` 执行变量替换。
2. 按“关键词 + 参数”规则解析目标插件。
3. 执行插件并得到结果列表。
4. 根据 `pick` 或默认规则选中结果，提取 `path` 作为下一步 `prev`。

## 6. 设置页交互设计

在 `SettingsWindow` 新增子页：`page_workflows`（标题“宏”）。

页面分区：

1. 宏列表区：显示 `id/name/description/steps 数`，支持选择、编辑、删除。
2. 编辑区：字段 `id`、`name`、`description`、`steps`（多行）。
3. 操作区：`新建`、`保存`、`取消`。

steps 输入格式：

- 一行一步：`command | pick`
- `pick` 可省略。
- 保存时转换为结构化 `steps` 数组。

交互规则：

1. 保存前执行完整校验并给出错误提示。
2. 删除宏需二次确认。
3. 保存后即时写入配置并可立即生效，不需重启。

## 7. 执行流程设计

入口：`WorkflowPlugin.handle_action(workflow_id_or_with_input)`。

流程：

1. 解析执行参数，拆分 `workflow_id` 与可选 `input` 文本。
2. 根据 `workflow_id` 查找宏定义。
3. 初始化执行上下文：`clipboard`、`prev`（空）、`input`（来自执行参数）。
4. 逐步执行 `steps`：
   1. 变量替换。
   2. 解析命令并定位插件。
   3. 调用插件 `execute()`。
   4. 根据 `pick`/默认规则选中结果。
   5. 将选中结果 `path` 写入 `prev`。
5. 全部成功后复制最终 `prev` 到剪贴板。
6. 返回成功提示：`工作流完成：<name>（共 N 步）`。

## 8. 错误处理策略

失败即中断，返回清晰错误原因：

1. `workflow_id` 不存在。
2. 宏结构非法或 steps 为空。
3. step 变量替换后命令为空。
4. 命令无法匹配插件关键字。
5. 插件执行返回空列表。
6. `pick` 未命中任何结果。
7. 选中结果无 `path`。
8. 剪贴板不可读/不可写。

错误信息采用“第 N 步 + 原因”格式，便于定位：

- 例如：`工作流失败: 第2步未匹配到结果（pick=MD5）`

## 9. 兼容性与迁移

1. 保留内置 4 个预设宏作为默认数据源（并直接写入 `DEFAULT_CONFIG.workflows`）。
2. 若配置中不存在 `workflows` 或数据非法，回退到默认预设并在保存时修复。
3. 搜索窗口 `workflow_run` 渲染与行为保持不变，避免额外改动。

## 10. 测试方案

### 10.1 单元/逻辑测试重点

1. 配置读写：`workflows` 默认值、合并、非法回退。
2. 引擎：变量替换、插件解析、`pick` 选择、异常中断。
3. 设置校验：`id` 正则、唯一性、step 语法。

### 10.2 手工回归场景

1. 新增宏并保存，重新打开设置页数据仍在。
2. `wf` 可检索到新增宏，预览描述正确。
3. 宏执行成功后最终结果进入剪贴板。
4. 故意配置错误 `pick`，能收到可定位错误。
5. 删除宏后二次搜索不再出现。

## 11. 变更范围

预计涉及：

1. `src/core/config.py`（新增 `workflows` 默认配置与读写兜底）
2. `src/plugins/workflow_tool.py`（配置化宏 + 命令链执行引擎）
3. `src/ui/settings_window.py`（新增宏管理页与表单校验）

## 12. 验收标准

1. 设置页可完成宏的新增、编辑、删除。
2. 新增宏可立即通过 `wf` 搜索并执行。
3. 命令链支持 `{clipboard}` 和 `{prev}` 变量联动。
4. 失败时可给出“第 N 步 + 原因”的错误提示。
5. 不影响现有非宏功能（搜索、插件、快捷键、截图等）。
